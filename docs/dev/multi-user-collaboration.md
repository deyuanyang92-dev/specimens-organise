# 多人协作操作手册（v0.5.0+ M1-M5）

支持 2-5 人小团队离线录入照片+标本，主管在中心机一键合并所有人的成果。

**核心承诺**：
- 任何旧版本工作区（v0.5.0 之前的 YZZ 单系列工作流）**零改动直接可用**，
  既可作为录入员的工作区上交，也能在中心机被聚合按钮吃下。
- 升级到"多人协作格式"只加两个配置键，不动任何 Excel / 照片文件，
  自动 snapshot 可一键回退。

    选保存位置 → 出 zip          到主管 incoming/             - 强制 snapshot
 2. 把 zip 通过 U 盘/邮件                                    - 段守护（manifest 内
    /共享盘 发给录入员                                         的 voucher_range 校验）
                                                              - import_workspace
                                                                （指纹冲突检测）
                                                              - 跨 voucher 同 SHA256
                                                                照片 → duplicates/

任意环节可以**绕过**：录入员若用旧版应用，跳过步骤 1-3，把整工作区拷给主管即可。
主管照样能在步骤 6-7 聚合。任务包只是为了"零撞号 + 段守护"。

## 主管视角

### 给录入员发任务包

1. 打开中心机工作区，菜单 **编号 → 批量生成编号…**
2. 填：
   - 生成数量（如 50）
   - 领取人（如 "张三"）
   - 用途、备注
3. 勾选两个 checkbox：
   - **「为该录入员使用独立编号前缀」** — 让张三的号都以 `ZS-` 开头
   - **「同时生成任务包 zip」** — 直接产出可发的 zip
4. 录入员前缀：填 `ZS-`（只允许 ASCII 字母/数字/横线/下划线）
5. 「生成并打包」→ 选保存位置 → 出 `任务包_张三_50个_ZS.zip`
6. 通过 U 盘 / 邮件 / SMB outbox/ 发给张三

中心机变化：
- `accession_series` 新增 `{name: "张三_系列", prefix: "ZS-", next_counter: 51, ...}`
- `编号分发记录.xlsx` 加一行「任务开始」（含 task_id + 编号段 ZS-000001..ZS-000050）

### 聚合所有人的成果

1. 主管把所有录入员的"工作区目录"统一放进 `incoming/`：
   - 走任务包的：录入员发回的整目录 / 服务器 incoming 上传的子目录
   - 不走任务包的（旧版 / 临时合作）：直接把对方工作区目录复制进来
2. 菜单 **工具 → 从收件箱聚合…**
3. 选 incoming 目录 → 输管理密码 `123`
4. 应用自动：
   - 创建合并前快照（`数据/数据版本/v....`）
   - 循环跑每个子目录（带 `manifest.json` 的优先按 packed_at 排序）
   - 段守护：voucher 必须落在 manifest 声明的 voucher_range 内（无 manifest 跳过此校验）
   - 指纹冲突检测（已有机制）
   - 跨 voucher 同 SHA256 照片 → 不写入，记录到 `duplicates/` 待审核
5. 完成弹消息：N 成功 / M 冲突 / K 错误 / D 重复疑似

子目录被分流到：
- `incoming/processed/` — 成功合并
- `incoming/conflicts/` — 指纹冲突（含冲突报告 xlsx）
- `incoming/errors/` — 超段 / 损坏 / 其它错（含 error.log）
- `incoming/duplicates/` — 跨 voucher 同 SHA256 审核报告 xlsx

## 录入员视角

### 收到任务包

1. 任务包是一个 zip。解压到一个空目录（如 `D:\工作\张三标本\`）
2. 用应用打开该目录作为工作区
3. 启动后菜单看不到任务包对话框，但 **「新增入库编号」直接生成 `ZS-000001`**
   开始（任务包内已配置好该系列为活动系列）
4. 照常录入标本 + 照片
5. 录满 50 个或截止时间到 → 收工

### 上交结果

**方式 A（推荐）— 任意位置复制整目录**：
1. 关闭应用（释放 lock）
2. 把整个 `D:\工作\张三标本\` 目录复制到 U 盘 / 邮件附件 / 服务器 incoming/
3. 主管端接收即可

**方式 B（M3.5+ 暂未实现）— 应用内"打包工作区结果"按钮**：未来版本会加。

### 提示

- 录入超过预留段（如领了 50 个但录到第 51 个）→ 应用不会阻止你录入，
  但主管聚合时会卡在「段守护」→ 子目录进 `errors/`，主管会找你扩段
- 不要手动改 `数据/工作区配置.json` 的 `accession_series` 段，会让段守护误判
- 字段模版 / 照片归档 / 快照 等所有现有功能完全可用

## 旧版本工作区怎么办

### 情景 1 — 工作区里已经录了一段时间，想加入多人协作

1. 工具菜单 → **升级工作区到多人协作格式…**
2. 应用：
   - 自动 snapshot
   - 在 `工作区配置.json` 加 `multi_user_protocol_version: "1.0"` + 
     `legacy_yzz_segment: [1, N]`（N 是当前最大 YZZ 序号）
   - 不动任何 Excel / 照片
3. 之后这个工作区就可以作为中心机或录入员工作区参与协作

### 情景 2 — 录入员仍用旧版应用

完全可以。把整工作区目录复制给主管，主管「从收件箱聚合」走降级模式（无 manifest 也能合并）。
唯一限制：没有段守护保护，要靠主管/录入员自觉约定编号段。

### 情景 3 — 已升级想退回旧版

- 菜单 **工具 → 数据版本…**（已有功能）→ 选升级前那张快照 → 恢复
- 或者 **工具 → 降低工作区兼容版本…**（已有功能，密码门控）

## 文件复制 vs 服务器共享 决策树

```
是否所有人在同一办公室 LAN？
├─ 是 → 有 NAS / 服务器吗？
│   ├─ 是 → 推荐方式：服务器共享盘 + 录入员手动复制
│   │       工作区目录到 \\server\specimen\incoming\
│   │       优点：复制即上交，主管随时聚合
│   │       缺点：录入员需要会用 SMB（基本都会）
│   └─ 否 → U 盘 / 邮件 / 局域网共享文件夹
│           都可以，最后效果一样
└─ 否 → U 盘 / 邮件 / 微信 / 网盘下载链接
         主管手动把收到的目录放进 incoming/ 即可
```

**关键**：incoming/ 是主管自己电脑上的一个普通目录。**不需要**专用服务器、
不需要数据库、不需要外网。**有 SMB 就用 SMB，没有就 U 盘**。

## 故障处理

| 症状 | 可能原因 | 处理 |
|------|----------|------|
| 聚合时弹"加锁失败" | 别的合并员同时跑了聚合 | 等几分钟重试；检查 incoming/ 下是否有 `processing_*` 目录长时间未消失（卡死则手动改名回原名后重试）|
| 子目录进 conflicts/ | 同 voucher 不同内容 | 打开冲突报告 xlsx 看差异；联系录入员核对；若中心方对则保留中心、删 conflicts/ 下子目录 |
| 子目录进 errors/ 且 error.log 写"超出 manifest 预留段" | 录入员录超了 | 在中心机用「批量生成编号」给同一录入员同一前缀**扩段**（如再发 20 个），然后把 errors/ 下子目录改名回正常名重试 |
| 子目录进 errors/ 且 error.log 写其它异常 | 数据损坏 / xlsx 不合法 | 让录入员检查源工作区是否能在他自己机器上正常打开；不能则从快照恢复 |
| duplicates/ 下有报告 | 跨 voucher 同 SHA256 照片 | 打开报告 xlsx 看：是合理的"一对多"（同标本多角度复用张总览图）→ 把该照片手动 add_photo 到该 voucher；是真的重复入库 → 联系录入员核对 |
| 聚合后中心数据有问题 | 任何原因 | 工具菜单 → 数据版本 → 恢复到"聚合前快照" → 一键退回 |
| 录入员 U 盘损坏 | 物理问题 | 让录入员从自己机器再发一次；若机器也丢了，从中心 alloc_log 看他领过哪段，让别人重录 |
| 两个录入员撞号了 | 没用任务包 / 都用 YZZ 单系列 | 主管聚合时会因指纹冲突报告；建议建议改用"录入员独立前缀"后续避免 |
