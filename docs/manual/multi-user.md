# 多人协作快速卡（M1-M5 新增功能）

> v0.5.0+。完整版手册见 `docs/multi-user-collaboration.md`，本卡是单页速查。

## 角色

| 角色 | 谁 | 做什么 |
|------|----|----|
| 主管 | 持有中心机工作区的人 | 发任务 / 收回成果 / 合并 |
| 录入员 | 拿到任务包的人（1-N 人）| 离线录入标本 + 照片 → 交给主管 |
| 合并员 | 通常 = 主管 | 在中心机点聚合按钮 |

---

## 主管：发任务包给录入员

> 用 M2 前缀分人 + M3 任务包。

1. 打开中心机工作区
2. 菜单 **编号 → 批量生成编号…**
3. 填表：
   - 生成数量：例 50
   - 领取人*：例 "张三"
   - 用途 / 备注
4. **勾选两个 checkbox**：
   - ☑ 为该录入员使用独立编号前缀（多人协作时强烈建议）
   - ☑ 同时生成任务包 zip
5. **录入员前缀**：ASCII，例 `ZS-`、`LS-`、`A1-`
6. **「生成并打包」** → 选保存位置 → 得到 `任务包_张三_50个_ZS.zip`
7. 通过 U 盘 / 邮件 / 微信 / 服务器共享 发给张三

中心机记录：
- `accession_series` 多一条 "张三_系列"（前缀 `ZS-`，已分配 ZS-000001..ZS-000050）
- `编号分发记录.xlsx` 多一行「任务开始」
- 下次给张三再发 30 个 → 接续 ZS-000051..ZS-000080，不撞号

## 录入员：使用任务包

1. 收到 zip → 解压到空目录，例 `D:\工作\标本任务_张三\`
2. 打开 specimen-organise 应用 → 选刚解压的目录作工作区
3. 点工具栏 **「+ 新增入库编号」** → 自动生成 `ZS-000001`（不是 YZZ000001）
4. 照常录入管内编号 / 保存方式 / 采集地点 / 采集日期
5. 拖入照片到照片网格
6. 继续 → ZS-000002、ZS-000003...
7. 录满 / 截止时间到 → 收工

**注意**：
- 不要手动改 `数据/工作区配置.json` 的 `accession_series`
- 意外录到 ZS-000051 / 超段：仍可录但交回时会被段守护拦下 → 找主管扩段

---

## 录入员：把结果交给主管

**A. U 盘 / 邮件附件** — 关闭应用 → 整目录复制到 U 盘 → 给主管

**B. 服务器 / 共享盘** — 关闭应用 → 复制到 `\\server\specimen\incoming\张三_20260516_1430\`

**C. 网盘 / 微信** — 整目录压成 zip 发；主管解压到 incoming/ 即可

---

## 主管：合并所有人的成果

> 用 M1 聚合 + M4 段守护 + 照片审核。

1. 把所有录入员的工作区目录统一放进 `incoming/`：
   ```
   C:\incoming\
     张三_20260516_1430\
     李四_20260517_0900\
     ...（含未走任务包的旧工作区也行，走降级模式）
   ```
2. 菜单 **工具 → 从收件箱聚合…** → 选 `C:\incoming\` → 输管理密码 `123`
3. 应用自动：
   - 创建合并前快照
   - 锁住每个子目录（rename → `processing_*`）
   - 段守护：voucher 必须在 manifest 段内
   - import_workspace 做合并（指纹冲突检测）
   - 跨 voucher 同 SHA256 照片 → duplicates/ 审核
4. 完成弹消息：N 成功 / M 冲突 / K 错误 / D 重复疑似 / 快照名

子目录被分流到：
```
C:\incoming\
  processed\        ← 成功合并
  conflicts\        ← 指纹冲突（含冲突报告 xlsx）
  errors\           ← 超段 / 损坏（含 error.log）
  duplicates\       ← 跨 voucher 同 SHA256 照片审核报告 xlsx
```

## 主管：升级旧版工作区到多人协作格式

> M5。仅当旧工作区已录了一段时间，现在想加入多人协作时用。

1. 打开旧工作区
2. 菜单 **工具 → 升级工作区到多人协作格式…**
3. 点 **「是」**
4. 应用自动：
   - 创建升级前快照
   - 写两个标记键到 `工作区配置.json`：`multi_user_protocol_version: "1.0"` + `legacy_yzz_segment: [1, N]`
   - 写「upgrade_to_multi_user_protocol」操作记录
   - **不动任何 Excel / 照片 / 已录数据**

回退：操作记录 → 撤销；或数据版本管理 → 恢复升级前快照。

## 故障速查

| 现象 | 原因 | 处理 |
|------|------|------|
| 聚合提示"加锁失败" | 另一合并员同时跑了 | 等 5 分钟重试；`processing_*/` 长时间不消失 → 手动改名回原名 |
| 子目录进 conflicts/ | 同 voucher 不同内容 | 打开冲突报告 xlsx；联系录入员核对；中心方对则保留中心 |
| 子目录进 errors/ "超出 manifest 预留段" | 录入员超段 | 「批量生成编号」给同录入员同前缀**扩段**；errors/ 下子目录改回原名重试 |
| 子目录进 errors/ 其它异常 | 数据损坏 | 录入员检查源工作区能否打开；不能则从快照恢复 |
| duplicates/ 下有报告 | 跨 voucher 同 SHA256 | 打开 xlsx：合理一对多 → 主管手动 add_photo；真重复 → 联系录入员 |
| 聚合后数据有问题 | 任何原因 | 工具 → 数据版本 → 恢复到聚合前快照 |
| 录入员 U 盘损坏 | 物理 | 录入员重发；若机器丢则从 alloc_log 看他领的段重录 |
| 撞号（没用任务包）| 两人用 YZZ 单系列 | 聚合时冲突报告；建议改用任务包 + 前缀分人 |

---

## 最佳实践

- **永远勾**「为该录入员使用独立编号前缀」— 前缀分人是零撞号的根本
- 主管每周聚合 1-2 次，不要堆几月才合
- 录入员每天收工时备份整工作区目录到 U 盘 / 服务器
- 重要合并后保留 `incoming/processed/` 1 个月便于审计
- duplicates/ 报告要尽快人工审核，拖久了忘原始上下文
