# 合并 / 导入 — 4 种场景操作示例

> S1/S2/S3/S7 新功能落地后的实用速查。完整背景见 `docs/multi-user-collaboration.md`。

## 场景 1：他人结果目录直接放 incoming/（最简）

### 适用
2-3 人协作，主管在自己电脑设一个 `C:\incoming\`，录入员把整工作区目录拷过来。

### 目录结构
```
C:\incoming\
├── ydy\                    ← 张三的工作区目录（含 数据/ + 照片/）
│   ├── 数据\
│   └── 照片\
└── yss\                    ← 李四的工作区目录
    ├── 数据\
    └── 照片\
```

### 操作
1. 主管打开中心机工作区
2. 菜单 **工具 → 从收件箱聚合…**
3. 选 `C:\incoming\` → 输管理密码 `123`
4. 弹**预览**（S7 新增）：
   ```
   候选子目录：2
   预计新增 voucher：23
   预计重复跳过：0
   预计冲突：0
   预计照片新增：45
   预计跨 voucher 同 SHA256 照片：0
   预计同名不同内容照片：1     ← S3 提示
   ...是否继续？
   ```
5. 点「是」 → 应用自动 snapshot + 合并 + 分流

### 结果
```
C:\incoming\
├── processed\              ← ydy + yss 已并入中心，被移到这里
├── conflicts\              ← 若有指纹冲突
├── errors\                 ← 若有数据损坏 / 超段
├── duplicates\             ← 若有跨 voucher 同 SHA256 照片
└── name_conflicts\         ← S3 同名不同内容照片报告
    └── 同名照片冲突报告_20260516_143000.xlsx
```

## 场景 2：嵌套目录结构（S1 新支持）

### 适用
录入员给的目录里**再包一层**工作区子目录，例如 `张三_20260516/工作区A/` 才是真正的工作区。

### 目录结构
```
C:\incoming\
├── 张三_20260516\          ← 不是工作区，是包装层
│   └── 工作区A\            ← 这才是工作区（含 数据/）
│       ├── 数据\
│       └── 照片\
├── 李四_20260517\
│   └── 工作区B\
│       └── 数据\
└── 项目X\                  ← 三层嵌套也支持（深度 ≤ 3）
    └── 录入员Y\
        └── 工作区Z\
            └── 数据\
```

### 操作
跟场景 1 完全一样：菜单 → 从收件箱聚合 → 选 `C:\incoming\` → ...

应用会**递归扫**所有"含 `数据/` 子目录"的工作区，最多 3 层深。

### 内部细节
扁平化后的处理子目录名 = 用 `__` 拼接相对路径段：
- `张三_20260516/工作区A/` → `张三_20260516__工作区A`
- 处理中锁名 = `processing_张三_20260516__工作区A`
- 完成后归档到 `processed/张三_20260516__工作区A/`

## 场景 3：直接选源目录（S2 新增，不用 incoming/）

### 适用
主管不想"先把每人目录复制到统一 incoming/"，希望直接在文件管理器选源目录。

### 操作
1. 主管打开中心机工作区
2. 菜单 **工具 → 批量导入工作区目录…**
3. 弹 Dialog：
   ```
   ┌─────────────────────────────────────────────────┐
   │ 选择多个工作区目录（每个必须含 数据/ 子目录）   │
   │                                                 │
   │ ┌─────────────────────────────────────────────┐ │
   │ │ D:\ydy                                      │ │
   │ │ E:\yss                                      │ │
   │ │ F:\backup\张三_20260516\工作区A             │ │
   │ └─────────────────────────────────────────────┘ │
   │ [添加目录…] [移除选中]                          │
   │                                                 │
   │              [取消] [开始合并]                  │
   └─────────────────────────────────────────────────┘
   ```
4. 点「添加目录…」逐个把 `D:\ydy`、`E:\yss`、`F:\backup\...` 加进列表
5. 点「开始合并」→ 输密码 → 应用自动合并

### 与场景 1 区别
- **不需要** 先把目录复制到 incoming/
- **不需要** 目录在同一个父目录下
- 源目录**原样保留**，不会被移动到 `processed/`
- duplicates 报告写在中心机 `数据/duplicates_报告/`

## 场景 4：旧版本工作区升级 + 合并

### 适用
团队里有人用了几个月**v0.5.0 之前**的本软件，现在要加入多人协作流程。

### 操作

**录入员（用旧版录的）侧 — 完全不动**：照常用旧版应用工作。

**主管侧（在中心机）— 一次性升级**：
1. 打开自己的中心机工作区
2. 菜单 **工具 → 升级工作区到多人协作格式…**
3. 应用：
   - 创建升级前快照
   - 在 `工作区配置.json` 加 `multi_user_protocol_version: "1.0"` + `legacy_yzz_segment: [1, N]`
   - 不动任何 Excel / 照片 / 已录数据
4. 完成

**接收旧版工作区**：
1. 录入员把整目录拷给主管
2. 主管放进 `incoming/` 或直接用「批量导入工作区目录」选源目录
3. 应用自动识别 — **不要求**对方升级（降级模式：无 manifest 也能合并）
4. 合并完成。冲突报告 / duplicates 报告等机制全部生效

### 兼容性承诺
- 旧版工作区**不需要先升级**就能被中心机吃下
- 旧版应用打开升级后的中心机工作区：受 schema_version 拦阻，**不会**崩；如需让旧版能写，主管用「降低工作区兼容版本…」一键退回

## 故障速查（合并相关）

| 现象 | 原因 | 处理 |
|------|------|------|
| 预览显示 "segment_violation" | 录入员录超了 manifest 段 | 用「批量生成编号」给同录入员同前缀扩段，errors/ 下子目录改回原名重试 |
| 预览显示 "unreadable" | 子目录的 标本信息.xlsx 损坏 / 不存在 / 表头不对 | 检查源工作区能否打开；不能则从快照恢复 |
| 预览显示 "conflict" | 同 voucher 不同内容 | 继续合并 → conflicts/ 报告 → 人工核对 |
| name_conflicts/ 下有报告 | 跨子目录同名不同内容照片 | 物理已加 _2 后缀避免覆盖，报告供主管事后核对 |
| duplicates/ 下有报告 | 跨 voucher 同 SHA256 照片 | 看 xlsx：合理一对多 → 主管在中心机手动 add_photo；真重复 → 联系录入员 |
| 合并后想退回 | 任何原因 | 工具 → 数据版本 → 恢复到聚合前快照 |

## 常用菜单一览（M1-M5 + S1-S7 + A1/A2）

| 菜单项 | 触发功能 | 引入版本 |
|--------|----------|----------|
| 工具 → 从收件箱聚合… | M1 + S1 嵌套 + S3 报告 + S7 预览 | M1, S1/S3/S7 |
| 工具 → 批量导入工作区目录… | 直接选源目录合并（不需 incoming）| S2 |
| 工具 → 升级工作区到多人协作格式… | 给旧工作区贴标记 + snapshot | M5 |
| 工具 → 导出 Darwin Core Archive… | 一键打 DwC-A zip（对接 GBIF）| A1 |
| 工具 → 从 EXIF 批量回填采集日期… | 自动从照片 EXIF 补"采集日期"空字段 | A2 |
| 编号 → 批量生成编号…（含「同时生成任务包 zip」勾选）| 给录入员发任务包 | M2 + M3 |
| 工具 → 降低工作区兼容版本… | 旧版应用兼容退路 | v0.5.0 |
| 工具 → 数据版本…（恢复版本）| 一键回退到任意快照 | v0.4.x |

合并 / 升级前**都自动 snapshot**，**所有改动可回退**。
